Bienvenidos a este desafío integrador. En la industria de los E-sports profesionales, la integridad de la información y la alta disponibilidad son pilares críticos. 2. FASE 1: DISEÑO Y PERSISTENCIA DE DATOS CON SQL (SUPABASE)

2.1. Modelado Relacional y Normalización

Un diseño de base de datos deficiente conlleva redundancia y pérdida de integridad. Aplicaremos rigurosamente las tres primeras formas normales:

1. Primera Forma Normal (1FN): Atributos Indivisibles. Siguiendo la lógica de "Ana Pérez", no podemos tener múltiples valores en una celda (como una lista de productos en un solo campo). En la tabla jugadores, cada campo debe ser atómico (ej. un solo email por registro).
2. Segunda Forma Normal (2FN): Dependencia Funcional Completa. Los atributos no clave deben depender de la totalidad de la clave primaria. En jugadores, el nombre depende del id, no de una relación parcial. Separamos las entidades en tablas distintas (equipos y jugadores) vinculándolas mediante claves foráneas.
3. Tercera Forma Normal (3FN): Eliminación de Dependencias Transitivas. Un atributo no debe depender de otro que no sea la clave primaria. Por ejemplo, en la tabla partidas, el nombre_equipo no debe estar presente; solo el equipo_id. El nombre del equipo depende del ID del equipo, y este a su vez de la clave de la partida. Incluir el nombre crearía una dependencia transitiva que debemos evitar.

2.2. Definición de Estructuras (DDL)

Para un sistema de alta concurrencia como un torneo global, utilizaremos BIGINT para las claves primarias para prevenir desbordamientos de datos.

-- Creación de la tabla de Equipos
CREATE TABLE equipos (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    nombre VARCHAR(50) NOT NULL UNIQUE,
    fecha_registro TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);

-- Creación de la tabla de Jugadores
CREATE TABLE jugadores (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    nombre VARCHAR(50) NOT NULL,
    equipo_id BIGINT REFERENCES equipos(id),
    email VARCHAR(100) UNIQUE NOT NULL
);

-- Creación de la tabla de Partidas
CREATE TABLE partidas (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    equipo_local_id BIGINT REFERENCES equipos(id),
    equipo_visitante_id BIGINT REFERENCES equipos(id),
    fecha_partida TIMESTAMP WITH TIME ZONE NOT NULL,
    resultado VARCHAR(50) DEFAULT 'Programada',
    CONSTRAINT check_equipos_distintos CHECK (equipo_local_id <> equipo_visitante_id)
);

3. FASE 2: CONTROL DE VERSIONES Y COLABORACIÓN (GIT & GITHUB)

3.1. Preparación del Entorno (Starter Kit)

Git no es solo una herramienta de seguimiento; es un sistema que captura Snapshots (fotos del estado) de su proyecto de forma Inmutable. Su flujo de trabajo se divide en:

1. Working Directory: Su área de trabajo local.
2. Staging Area: El "limbo" donde seleccionan cambios granulares. Su existencia es clave para crear commits limpios y temáticos, evitando mezclar cambios de lógica con cambios estéticos.
3. Repository: El historial definitivo de commits.

Comandos:

* git init: Inicializa la base de datos interna en .git/.
* git status: Diagnóstico esencial para ver archivos modificados o en staging.

3.2. Gestión de Cambios y Ramas Estéticas

El desarrollo profesional prohíbe trabajar directamente sobre main.

1. Cree su rama de trabajo: git checkout -b feature/estetica.
2. Tras modificar el CSS/HTML, use git add . para mover al Staging Area.
3. Confirme con: git commit -m "feat: optimización de la interfaz responsiva del tablero de resultados".

3.3. Sincronización con GitHub y Autenticación Segura

Como estándar de seguridad industrial, utilizaremos criptografía de par de llaves.

Característica	Tokens de Acceso Personal (HTTPS)	SSH Keys (Recomendado)
Seguridad	Tokens revocables con permisos limitados.	Criptografía asimétrica (Clave Privada/Pública).
Experiencia	Requiere ingresar token periódicamente.	Transparente tras la configuración inicial.
Algoritmo	Basado en contraseñas temporales.	ed25519 (Moderno, seguro y corto).

Configuración SSH Paso a Paso:

1. Generar llave: ssh-keygen -t ed25519 -C "su_email@ejemplo.com".
2. Obtener clave pública para GitHub: cat ~/.ssh/id_ed25519.pub.
3. Probar conexión: ssh -T git@github.com.

Vinculación: git remote add origin <URL_SSH_DEL_REPOSITORIO> git push -u origin main

4. FASE 3: DESPLIEGUE EN LA NUBE (MICROSOFT AZURE)

4.1. Organización de Recursos en Azure

Las decisiones arquitectónicas sobre latencia impactan directamente en la experiencia de usuario (UX).

* Jerarquía de Azure: Identidad (Cuenta) > Facturación (Suscripción) > Contenedor Lógico (Resource Group) > Servicio (Recurso).
* Decisión: Crear un Resource Group llamado rg-esports-tp en la región Brazil South para minimizar el round-trip time de los paquetes de datos.

4.2. Despliegue de la Static Web App

Azure Static Web Apps distribuye su contenido mediante una CDN global, asegurando tiempos de carga mínimos y proveyendo un certificado SSL (HTTPS) automático. Desde la extensión Azure Tools en VS Code:

1. Inicie sesión y elija "Create Static Web App".
2. Seleccione su Suscripción y el Resource Group rg-esports-tp.
3. Al finalizar, Azure generará una URL pública donde su sistema estará vivo.

4.3. Configuración de Seguridad y Conectividad (Variables de Entorno)

Este es el paso más crítico del despliegue. Bajo ninguna circunstancia deben exponerse las credenciales de Supabase en el código fuente.

1. Acceda al Portal de Azure > Su Recurso > Configuration (o Application Settings).
2. Defina las variables (ej. SUPABASE_URL, SUPABASE_KEY).
3. Esto constituye un protocolo de seguridad obligatorio para prevenir filtraciones de credenciales en repositorios públicos.
